## 一、已完整实现的功能

### 1.1 核心工作流 ✅

5 阶段订单流水线完整运行：
`accept (12s)` → `execute (18s)` → `review (14s)` → `acceptance (10s)` → `settlement`

审查阶段骰子机制已实现，结果三分支（通过/打回/冲突）均正常工作。  
骰子权重受组织原型（`reviewStrength`）和今日策略（`reviewApproveDelta`）双重影响。

### 1.2 订单系统 ✅

4 种订单类型全部实现，奖励区间与 GDD 基本对齐：

| 类型 | GDD 范围 | 实现范围 |
|------|---------|---------|
| hotspot | ¥4k~17k | ¥6k~15k |
| brand | ¥18k~32k | ¥20k~32k |
| app | ¥13k~24k | ¥15k~22k |
| mystery | ¥33k~52k | ¥35k~50k |

每日刷新 2~4 个，生命周期状态机完整。

### 1.3 老板技能系统 ✅

4 个技能全部实现，冷却/费用/次数限制与 GDD 完全一致。  
强制拍板中断工作流机制（`workflowAutoAdvance_` 标志）正常工作。

### 1.4 私下频道（3 个 MVP） ✅

| 频道 | 解锁条件 | GDD 是否一致 |
|------|---------|------------|
| 工部摸鱼小群 | 工部连续 3 天高压（≥45% 工作量） | ✅ |
| 蛐蛐老板实名群 | 强制拍板 ≥ 3 次 | ✅ |
| 工部门下省秘密结盟 | 部门冲突 ≥ 5 次 | ✅ |

懒加载机制已实现（首次点进频道才生成内容）。  
只读限制正确（老板无法发言，`readonly = true`）。

### 1.5 LLM 双模式架构 ✅

- **simulate 模式**：`DialoguePool` 按工作流阶段 × 订单类型组织，覆盖完整
- **api 模式**：C/S RemoteEvent 代理完整，失败时自动降级到预设兜底
- 验收官 JSON 结构化解析（`passed/score/reason`）正常工作

### 1.6 UI 架构 ✅

竖屏企业内网风格完整实现：
- 顶栏（公司状态 + 资金声誉）
- Tab 四栏（消息/工作台/指挥/财务）
- 频道列表 Overlay（公开 + 私下分组，未读计数）
- 聊天面板（按频道切换，自动滚底）
- 指挥面板（技能卡片 + 部门状态）

## 二、部分实现（与 GDD 有差距）

### 2.1 事故系统 ⚠️

**GDD 目标**：30 个事故，每种组织结构至少 5 个专属奇观  
**当前实现**：4 个通用事故，无组织专属绑定

已实现的 4 个：
- `approval_loop` — 审批循环（中书省/门下省意见互相矛盾）
- `quality_meltdown` — 质检熔断（门下省打回停摆）
- `zhongshu_coup` — 中书省越权
- `missed_hotspot` — 热点过期

触发机制为概率驱动（冷却 45s），但触发概率较低（4%~7%），实际游玩中不够频繁。  
GDD 强调「奇观事故是截图传播素材」这一设计意图尚未充分体现。

### 2.2 验收官系统 ⚠️

**GDD 目标**：4 种订单对应 4 种不同性格的验收官
- 热点单 → 急性子网红编辑
- 品牌单 → 挑剔品牌总监
- 应用开发单 → 务实产品经理
- 神秘单 → 身份不明甲方

**当前实现**：验收官为单一角色，`AcceptanceAgent` 中仅有通用提示词，未按订单类型切换 system prompt。  
JSON 结构化解析功能完整，但角色差异化未实现。

### 2.3 组织生成引擎 ⚠️

**GDD 目标**：玩家输入一句话自然语言（如「唐朝三省六部」）自动生成架构  
**当前实现**：4 种原型硬编码在 `OrgGenerator`，游戏开始时随机选取，无用户输入界面  
自然语言 → 原型匹配的入口未实现。

### 2.4 部门 AI 性格标签 ⚠️

**GDD 目标**：每个 AI 单位携带 1~3 个性格标签，标签影响发言风格  
（过度思考/上下文遗忘/极度自信/容易幻觉/爱甩锅/懂合规/审美优等）

**当前实现**：`AgentProfiles` 中有固定性格描述，但标签为静态设定，不可动态组合，无运行时随机分配机制。

### 2.5 私下频道（GDD 5 个，已实现 3 个）⚠️

**未实现的 2 个**：
- 「中书省小算盘」（解锁条件：中书省升级到 2 级）— 当前无部门升级系统
- 「验收官吐槽群」（解锁条件：出现过验收不通过）— 解锁逻辑未在 `SecretChannelSystem` 中

### 2.6 分享卡系统 ⚠️

**GDD 目标**：组织图海报、周结算战报、带水印事故截图、私下频道截图（图片格式）  
**当前实现**：`ShareCardSystem` 仅生成文本战报推送到全局频道，无图片生成、无截图导出、无水印处理

## 三、未实现功能

### 3.1 大循环（月度） ❌

GDD 定义了月度大循环：市场环境变化 → 调整组织 → 承接更复杂订单 → 危机 → 升级  
当前游戏为无限日循环，无月度节点、无市场变化、无组织升级路径。

### 3.2 高级 Agent 单位 ❌

GDD V1.1 规划的「尚书省（调度 AI）」、「审计官」、「热点预警官」均未实现。

### 3.3 商业化模块 ❌

激励广告、IAP、皮肤包、永久解锁私下频道等均未实现。

### 3.4 财务 Tab ❌

UI 中财务 Tab 为占位符，无实际内容（收支明细、趋势图等）。

### 3.5 今日策略选择 ❌

GDD 中每天开始时玩家可设定「今日策略方向」。  
`GameConfig` 中有 `DAILY_STRATEGIES` 定义，`StatusBar` 有 chip 显示，但实际切换入口和策略选择 UI 未完整实现。

## 四、与 GDD 的关键偏差

| GDD 设计意图 | 当前实现差异 |
|------------|------------|
| 「不是数值模拟，而是真实多 Agent 工作流」 | simulate 模式下退化为预设模板，API 模式才体现设计意图 |
| 验收官不受老板控制，是独立裁判 | 机制上正确，但单一角色设定削弱了甲方差异感 |
| 私下频道情报可转化为决策依据 | 频道内容存在，但无对应的「反制行动」UI 入口 |
| 奇观事故是天然截图传播素材 | 事故数量不足，触发概率低，传播效果未验证 |
| 一句话生成 AI 公司 | 核心差异化特性，当前完全为静态选择 |

## 五、技术健康度

### 优点
- 模块化良好，EventBus 解耦彻底，扩展成本低
- 双模式 LLM 降级机制完善，离线可正常游玩
- 消息队列 + 延迟推送机制使对话节奏自然
- 代码注释充分，类型标注规范

### 需关注
- 消息队列使用秒精度 dt 累加，长时间运行可能有误差漂移
- `DialoguePool` 随游戏扩展会成为维护瓶颈（当前已有大量硬编码模板）
- `LLMConfig.lua` 含 API Key，需确认未提交到版本控制
