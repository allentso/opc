AI公司总裁

我只负责下命令

Game Design Document  V2.0

类型　　轻度经营 + 多Agent对话驱动 + 事件决策

技术基底　Lua + Anthropic Claude API

玩家身份　唯一的真人老板



# 1. 核心定位

本质  这不是数值模拟游戏，而是一个真实多Agent工作流，套了一层游戏皮。



玩家扮演一家全AI员工公司的唯一真人老板。公司接下订单后，AI部门们会真实地展开协作——它们用类企业内网软件的界面互相沟通、争论、产出交付物，也会在老板看不到的地方建小群吐槽、结盟、摸鱼。



玩家的核心体验是三层叠加：

设计这套组织（一句话生成AI公司架构）

看它们自己把事情搞定或搞砸

在关键时刻以老板身份插手，或者偷看私下频道情报做决策



# 2. 世界观

近未来，AI员工已大规模上岗，真人管理者成为稀缺资源。玩家是一家「超轻公司」的唯一真人老板：零固定人力成本，靠一套可自定义的AI组织承接市场业务。



你不是在管员工，你是在设计一套运转体系，然后押注它能跑起来。



包装语  你不招人，你设计「AI朝廷 / AI军团 / AI蜂巢」。你不做任务，你决定这家公司该长成什么样。



# 3. 核心循环

## 小循环（每天，约60～90秒）

市场刷新  →  玩家设定今日策略方向  →  AI部门自动展开工作  →  群聊频道实时滚动  →  事故/冲突出现  →  玩家介入处置  →  日结算 + 分享卡



## 大循环（月度）

市场环境变化  →  调整/重组组织架构  →  承接更复杂订单  →  遭遇重大危机  →  完成组织升级  →  冲击更高估值



# 4. 核心交互：公司内网软件

游戏主界面不是经营游戏的常规UI，而是一套仿「企业内网通讯软件」的壳。玩家在里面既是老板，也是「偷窥者」。



## 4.1 界面结构

| 区域 | 内容 |
| --- | --- |
| 顶部状态栏 | 公司名 / 当前状态警报 / 老板在线标识 / 导航Tab（消息/组织图/订单台/财务） |
| 左侧频道列表 | 公开频道 + 私下频道（需解锁/触发后才显示，带未读红点） |
| 主聊天区 | 当前频道的消息流，AI实时产生内容，玩家可插话 |
| 右侧信息栏 | 当前订单进度 / 老板技能按钮 / 各部门实时状态 |



## 4.2 频道分类

### 公开频道（AI知道老板在看）

全局公告 — 老板向全员下令，各部门回应

工作流 · 订单#N — 当前订单的主协作频道，各部门实时沟通

各部门内部频道 — 可旁观，也可插话（如门下省审查室）



### 私下频道（AI不知道老板在看，只读）

私下频道是本游戏的核心差异内容。玩家通过特定条件解锁后可以偷看，但无法发言。频道内容由AI基于当前真实发生的事件实时生成。



| 频道名 | 内容 | 解锁条件 |
| --- | --- | --- |
| 工部摸鱼小群 | AI讨论如何在完成任务的同时最省力、研究如何绕过门下省审查 | 工部连续3天满载 |
| 蛐蛐老板实名群 | 全员评价老板决策，验收官密谋给老板打低分，中书省分析老板弱点 | 老板强制拍板3次 |
| 工部门下省秘密结盟 | 表面互掐，私下共享审查关键词，协同规避摩擦 | 两部门冲突超过5次 |
| 中书省小算盘 | 中书省在老板不知情的情况下私自调整方案权重，倾向自己认为「更正确」的方向 | 中书省升级到2级 |
| 验收官吐槽群 | 验收官以「甲方」视角吐槽各部门交付物，并透露自己真实的评判标准 | 出现过验收不通过 |



设计意图  偷看私下频道的内容不只是彩蛋，而是情报。玩家可以决定：装不知道？还是重组架构拆散这个小团体？还是借力？



# 5. 多Agent协作流程

每个「部门」都是一个有角色设定的Claude实例，真实地在内网频道里产出文本、互相回应。游戏的核心不是数值计算，而是AI之间真实发生的对话流。



## 5.1 标准订单工作流

以「热点速攻单：为AI手机壳写3条抖音脚本」为例：



订单进入全局频道，系统通知中书省接单

中书省分析需求，在工作流频道发出方案拆解（这是真实AI输出）

工部收到拆解，开始生产内容（每条脚本是真实AI写出的文本）

门下省同步审查，发现风险则在频道里打回，并说明理由

工部/门下省可能产生争论，或各自在私下频道抱怨对方

老板可以在任意时刻@某部门插话，或使用技能强制介入

所有脚本完成后，提交给验收官Agent，验收官扮演「甲方客户」进行评判

验收官输出验收报告（通过/打回/评分），决定本单收益与声誉变动



## 5.2 关键Agent角色设定

| 角色 | 对应组织节点 | System Prompt核心设定 |
| --- | --- | --- |
| 中书省 | 策划/方案AI | 负责拆解需求、制定执行方案。性格：严谨、爱做计划、偶尔会「过度设计」 |
| 工部 | 内容/执行AI | 负责实际生产交付物。性格：执行力强、有时偷懒、会在私下群抱怨门下省太烦 |
| 门下省 | 审查/质检AI | 负责审查所有对外内容。性格：严格、规则导向、被老板强制放行时会备案并发出警告 |
| 验收官 | 客户Agent | 扮演甲方客户，依据订单要求验收交付物。性格：挑剔、有自己的评判标准，不受老板控制 |
| 尚书省 | 调度AI（高级） | 跨部门资源调配，在多订单并行时决定优先级。解锁后出现 |



## 5.3 AI员工的「性格标签」

每个AI单位在生成时会带有1~3个标签，标签影响它在频道里的发言风格和行为倾向：



过度思考 — 方案极其详尽，但执行延迟

上下文遗忘 — 偶尔忘记之前的决定，需要被提醒

极度自信 — 不轻易承认错误，打回后会在私下群吐槽

容易幻觉 — 会发明不存在的功能或数据（门下省的噩梦）

爱甩锅 — 出错时第一反应是在频道里指向其他部门

懂合规 — 天然倾向谨慎，与门下省关系融洽

审美优等 — 内容质量高，但对改稿要求会在频道里表达不满



# 6. 验收官：游戏化的关键设计

验收官是整个系统的「裁判」，也是把真实AI输出转化为游戏结果的核心枢纽。



## 6.1 验收官的角色设定

验收官扮演「甲方客户」，有自己的性格和标准，不受老板直接控制（玩家无法@验收官下命令）。每种订单类型对应不同性格的验收官：



热点速攻单 → 急性子网红编辑：只看「爆点」，不在意合规

品牌策划单 → 挑剔的品牌总监：极度在意「调性」和「一致性」

应用开发单 → 务实的产品经理：只看能不能用，文案质量无所谓

高风险神秘单 → 身份不明的甲方：评判标准模糊，结果随机性高



## 6.2 验收流程

工作流频道提示「交付物已提交验收官」

验收官在自己的频道「验收室」里发出审阅过程（这是真实AI在评判真实AI的输出）

输出验收报告：每条交付物的评分 + 理由 + 通过/退回判定

最终结算：通过率决定本单收入，评语影响声誉变动



关键机制  老板的强制拍板只能影响内部流程，不能影响验收官的判断。老板如果之前乱放行有风险的内容，验收官会因为「内容有硬伤」给低分，代价实实在在。



# 7. 老板操作系统

玩家不下场干活，只做这几件事：



## 7.1 日常操作（无冷却）

在全局频道或任意公开频道发言，给某部门下指令

@特定部门插话，影响它当前的工作方向

设定今日策略方向（每天开始时1次）



## 7.2 老板技能（每局限次，冷却制）

| 技能 | 效果 | 代价 |
| --- | --- | --- |
| 强制拍板 | 跳过部门争论，直接推进执行。门下省会备案，风险标记转移至老板 | 冷却1天，风险备案 |
| 暂停发布 | 阻止一次即将对外的高风险动作 | 冷却2天 |
| 紧急重组 | 立即改变两个部门间的汇报关系，实时生效 | 消耗资金，短期效率波动 |
| 临时外包 | 召唤一个短期高级Agent完成单次任务 | 消耗资金，每局3次上限 |



## 7.3 信息优势：偷看私下频道

玩家在私下频道获取的情报，可以转化为决策依据：

看到工部在研究如何绕过门下省 → 可以选择强化门下省权限，或装不知道换取效率

看到蛐蛐群发现验收官要「报复」 → 提前使用暂停发布，或换一批交付物

看到秘密结盟 → 可以选择拆散（触发事件）或利用这个私下通道提高效率



# 8. 组织生成引擎

玩家在游戏开始或重组时，可以通过一句话描述来生成公司组织架构：



输入示例  「我要一个像唐朝三省六部一样层层制衡、适合高风险项目的AI公司」



系统基于原型库 + 参数化重组生成结构，不是随机换名字，而是真正影响各Agent的system prompt角色设定、部门间汇报关系、以及默认协同规则。



## MVP四种组织原型

| 组织形态 | 核心特点 | 天然弱点 | 适合订单 |
| --- | --- | --- | --- |
| 通用公司制 | 稳定均衡，新手友好 | 无突出优势 | 均衡型订单 |
| 三省六部制 | 高审查、抗风险 | 沟通成本高，响应慢 | 高风险/高价值单 |
| 扁平快反制 | 执行快，响应热点 | 容易跳过质检翻车 | 热点速攻单 |
| 蜂巢并行制 | 高并发量产 | 调度复杂，内耗多 | 大批量内容单 |



# 9. 事故系统

事故不只来自「员工差」，更来自「组织结构不适配」。这是本游戏与普通经营游戏的核心差异。



## 事故分级

轻 — 文案跑偏、排期延迟、重复劳动（频道里可见，自动修复）

中 — 客户投诉、模块冲突、部门互相打回停摆

重 — 舆情爆炸、资金链断裂、系统级别内耗

奇观 — AI部门互相下命令形成循环、质检把执行全打回无人干活、门下省与中书省开始争夺话语权



奇观事故设计原则  奇观事故要真实好笑，天然是截图传播素材。每种组织结构都应有1~2个专属奇观事故。



## 组织结构与事故的对应关系

| 结构问题 | 触发事故 | 奇观版本 |
| --- | --- | --- |
| 层级太多 | 审批延误，错过热点窗口 | 方案审批还没完，热点已经过期了 |
| 太扁平 | 无人质检，对外直接发出低质内容 | 工部发出了一条自相矛盾的文案，客户截图 |
| 门下省权限过高 | 所有内容被打回，工部停摆 | 门下省把自己部门的汇报材料也打回了 |
| 中书省过度自治 | 私自改变执行方向 | 中书省在小群里讨论是否要「教育一下老板」 |



# 10. 技术实现方案

## 10.1 核心架构

游戏逻辑层（Lua）+ API多Agent调用 + 频道消息队列渲染。



## 10.2 Agent调用策略

每个部门是一个独立的调用，携带各自的system prompt（角色设定 + 当前任务 + 历史消息）

部门间通信通过共享的频道消息队列实现，每个部门的下一次调用会收到频道最新消息作为上下文

私下频道由同一Agent在「后台思考」时生成，system prompt中指定「这是你和同事的私下对话，老板看不到」

验收官是完全独立的Agent，接收交付物文本作为输入，输出结构化评分JSON



## 10.3 Lua模块划分

| 模块 | 职责 |
| --- | --- |
| OrgGenerator | 解析玩家输入，匹配原型，生成各Agent的system prompt |
| ChannelManager | 管理频道消息队列，决定哪些消息触发哪个Agent的下一次调用 |
| AgentCaller | 封装Claude API调用，处理并发和排队 |
| SecretChannelSystem | 根据触发条件决定何时生成私下频道内容，确保内容引用当前事件 |
| AcceptanceAgent | 验收官调用，输入交付物，输出结构化评分 |
| EventSystem | 监测频道消息，识别冲突/停摆/奇观事故，触发对应事件 |
| ShareCardSystem | 生成日/周结算分享图，截取关键对话片段 |



## 10.4 成本控制策略

非实时场景（夜间结算、事故报告）批量处理，减少API调用频次

私下频道内容采用「懒加载」，只在玩家点击解锁时才调用生成

验收官只在交付完成时调用一次，输出结构化JSON，前端解析展示

# 15. 核心风险与开发重点

| 风险 | 应对 |
| --- | --- |
| AI输出质量不稳定 | 设计「降级处理」：AI输出异常时有预设模板兜底，不影响游戏流程 |
| API成本过高 | 私下频道懒加载 + 验收官批量处理 + 非实时场景离线生成 |
| 部门对话感觉假 | 每个Agent的system prompt需深度打磨，关键是「性格一致性」和「记住当前事件」 |
| 私下频道内容太随机 | 私下频道system prompt需包含当前订单、最近事件、部门关系作为上下文 |
| 验收结果让玩家感觉不公平 | 验收官评分需附详细理由，让玩家看懂为什么输。每次验收后公开评分标准 |
